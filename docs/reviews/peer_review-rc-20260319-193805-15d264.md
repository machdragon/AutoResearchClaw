# Stage 18: Peer Review

### Reviewer A (Methodology / Systems)

**Strengths**

- Clear, well-motivated objective tightly tied to the Parameter Golf lane_3 setting (int8+zlib roundtrip, 16MB artifact cap, quick-gate).
- Method section is mathematically precise: formalizes `val_bpb` post-roundtrip, defines operators \(Q, S, C, D\), and states an explicit constrained optimization problem.
- Staged training design (float → QAT → QAT+codec-gates) is conceptually coherent and aligned with the stated hypothesis: aligning training to the int8+zlib roundtrip while respecting a gate budget.
- Baselines are thoughtfully defined to isolate axes of interest (no QAT vs QAT; no codec vs codec-in-loop; naive vs entropy-shaped layouts).

**Weaknesses**

1. **Topic alignment / overclaim vs actual evidence**

   - The paper is positioned as if ZQAT is a fully executed empirical method, but the “Results” section explicitly has no numbers, and the provided run evidence shows:
     - Only a single attempted run.
     - Run crashes before logging metrics (`val_bpb_roundtrip` missing; stack trace).
     - Refinement summary: “converged=false, stop_reason=consecutive_no_metrics, best_metric=null.”
   - Yet the abstract, intro, and discussion repeatedly speak as if experiments *will* show improvements. This is still a methodology proposal, not an executed study.

   **Actionable revision**
   - Reframe the entire paper as a *methodology and infrastructure* paper, not an empirical comparison. Replace all future-tense performance claims with accurate statements about what is currently implemented vs what remains.
   - Add an explicit statement in the introduction and conclusion that **no empirical validation of the main hypothesis has been completed**, and that the work is limited to framework design and integration.

2. **Hypothesis and topic adherence**

   - The core topic is “Parameter Golf val_bpb minimization — lane_3: QAT + roundtrip-aware compression alignment, hypothesis: staged QAT aligned to int8+zlib roundtrip improves post-roundtrip val_bpb vs float-only under same gate budget.”
   - Most of the paper respects this. However:
     - Several digressions into carbon/energy and general deployment/safety (e.g., Chien/Mariette, Yang/Mehmood) are tangential and not linked back to lane_3 constraints or val_bpb.
     - Some “Deployment-aware training” subsection text reads more like generic environment advocacy than a contribution directly tied to the proposed QAT/codec method.

   **Actionable revision**
   - Trim or move to a short “Broader Impacts” paragraph the passages on carbon footprint, runtime verification, and generic edge-latency work unless you explicitly connect them to concrete design decisions in ZQAT (e.g., choice of gate budget, run-time logging).
   - Ensure every paragraph in Intro/Discussion explicitly mentions Parameter Golf lane_3 / `val_bpb` / int8+zlib if it is intended as a mainline argument; otherwise, demote it or remove.

3. **Method–implementation mismatch**

   - Method section claims:
     - Per-channel affine int8 scheme.
     - Staged fake-quant schedule.
     - Gate budget \(B\) controlling explicit codec calls at specific steps.
   - Actual code snippet shows:
     - A *single* set of global QAT hyperparameters; no explicit gate schedule.
     - `compression_roundtrip_enabled: True` but no visible gating logic or mapping from the “gate budget” concept to implemented behavior.
     - Multi-seed support is declared (`"seeds": [42, 123, 456]`) but only one run was executed and crashed.

   **Actionable revision**
   - Add a concrete subsection **“Implementation Status vs Design”** where you:
     - Map each claimed algorithmic component (stages, gate schedule, entropy-shaped layouts) to specific config flags or functions in the code.
     - Clearly state which pieces are implemented and verified vs which are still planned.
   - If gate scheduling is not yet implemented (only a boolean `compression_roundtrip_enabled`), either:
     - Implement the actual step-scheduling and log the gate times in the run, or
     - Downgrade claims to “we plan to add a gate-budgeted schedule; current prototype only turns roundtrip on/off.”

4. **Completeness and section depth**

   - Word count: the body appears to be in the right range, but:
     - Method and Experiments are deep and precise; Results is essentially empty and cannot support any conclusions.
     - A full NeurIPS-style paper at this length with **zero data** is incomplete.
   - The “Parameter-Golf-compliant evaluation protocol” is described in detail, but many specifics are left as “once runs complete, we will…” — this weakens the methodological contribution.

   **Actionable revision**
   - Either:
     1. Substantially shorten the paper into a **short / workshop paper** that explicitly pitches ZQAT as an in-progress system, or
     2. Delay submission until you have **at least**:
        - One successful run per method.
        - A completed primary table with real numbers.
