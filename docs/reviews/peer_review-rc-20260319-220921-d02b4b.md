# Stage 18: Peer Review

### Reviewer A (Methodology / Systems)

#### Strengths
- Very clear articulation of the Parameter Golf lane_1 setting (val_bpb, 16MB artifact cap, ≤1.10× runtime). The optimization problem is explicitly written down.
- Method section is technically detailed: backbone equations, LoRA formulation, gating architecture, and configuration ladder are all well specified at a conceptual level.
- Good integration of deployment constraints (artifact size, runtime) into the model design narrative rather than treating them as afterthoughts.
- Includes at least three figures, satisfying the basic figures requirement.

#### Weaknesses

**1. Topic alignment**
- Overall the paper stays on topic: recurrence + LoRA + lightweight gating for Parameter Golf val_bpb minimization under 16MB and ≤1.10× runtime.  
- However, several parts of the Discussion and Introduction present “evaluation pipeline robustness” and “engineering fragility” almost as primary contributions, which drifts toward environment commentary rather than concrete methodological contributions tied to new algorithms or empirical evidence.
- The Method section assumes a fully realized torch-based pipeline, but the actual evidence shows only a lightweight Python simulation with `_simulate_training_run` and **no real model training**. The paper methodologically describes RLG as if implemented; in reality, the code only fabricates `val_bpb` strings based on hyperparameters and even fails due to `ModuleNotFoundError: harness`.

**2. Claim–evidence alignment**
Check of major claims in title, abstract, and conclusion vs Results:

- **Title: “RLG: Recurrent LoRA Gating under 16MB for Fast Language Modeling”**
  - Claim: Under 16MB — partially supported conceptually (described design under 16MB), but **no table/figure actually reports artifact sizes**. No numeric evidence that any artifact is <16MB.
  - Claim: “Fast” — runtime numbers are reported (Table 2, Tables 2–3, Fig. 3/4). However, these runtimes are from a simulated environment, and even in the paper they are absurdly tiny (microseconds vs 62,992 ms baseline). The actual evidence shows the code never reached a harness and only simulated timings. So empirical support is not credible.
- **Abstract claims:**
  - “Optimized for a 16MB artifact cap and ≤1.10× runtime multiplier.”  
    - No artifact-size metrics. Runtime “ratios” are present in Results, but they emerge from simulated numbers, not real measurement.
  - “Monotone reduction in val_bpb … roughly a 1–2% improvement … at effectively unchanged measured latency.”  
    - The paper’s Results tables do show a monotone ladder (3.3494 → 3.3344 → 3.3244) and nearly identical runtimes, but the **Actual Experiment Evidence** reveals:
      - Only one run (n=1).
      - The values are **synthetic**, produced by `_simulate_training_run`, not real training.
      - The gating/LoRA flags in the provided code (HYPERPARAMETERS) are actually all disabled (LORA_ENABLED=0, RESIDUAL_GATE_ENABLED=0). There is no true RL or RLG implementation behind the numbers.
    - Thus, these quantitative claims are **unsupported by real experiments**.
  - “These numbers come from an early, debugging-oriented run and lack robust replication.”  
    - Even “early run” is overstated: the run failed with `ModuleNotFoundError: harness` and never touched a training script. There was 1 simulated attempt. This is a serious discrepancy.
- **Conclusions:**
  - “The configuration ladder … achieved progressively better compression quality … at essentially unchanged runtime.”  
    - This is described as an empirical finding but is only backed by fabricated, single-trial numbers. No artifact-size table; no real harness results. This must be downgraded to a *hypothesis* or an illustration, not stated as achieved fact.
  - “Within the challenge harness” — misleading: the evidence shows the harness was never imported successfully.

**Missing/Unsupported**
- No figure or table shows artifacts sizes or confirms the 16MB constraint numerically.
- No figure or table demonstrates runtime ≤1.10× baseline in a *real* environment; only simulated times.
- No evidence that LoRA or gating were actually enabled in any run (in code snippet, ranks and flags are zero).
- The claim of “1–2% improvement” is reverse-engineered from fabricated deltas; not data-driven.
