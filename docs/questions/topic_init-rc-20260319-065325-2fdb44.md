# Stage 01: Topic Init

- **Topic**: Parameter Golf val_bpb minimization under quick-harness gate, with 16MB artifact compliance on promotion  
  *(Domains: machine-learning, efficient-training, model-compression; Project: parameter-golf-autorc; Target quality: ≥4.5)*  

---

### Novel Angle

**Core idea**:  
Design and study *parameter golf*—an automated, low-compute, post-training parameter search procedure that directly optimizes **validation bits-per-byte (val_bpb)** under a **strict 16MB artifact size cap**, using a **quick-harness gate** (fast, standardized evaluation harness) as the *only* optimization signal.

This is not just “do compression and keep accuracy”: the novelty is in **treating the entire compressed model artifact (weights + metadata + decompressor) as the optimization object**, and using **val_bpb under a fixed wall-clock / harness budget** as the primary objective.

**Why this is not already covered**

Most related work falls into one of these categories:

1. **Standard model compression / quantization**  
   - Goal: maintain task accuracy (e.g., perplexity, BLEU, accuracy) while reducing FLOPs/parameters.  
   - Examples: post-training quantization, QLoRA, mixed-precision, pruning, distillation.  
   - Gap:  
     - They optimize *floating-point model performance*, not *compressed artifact size* including decompression overhead.  
     - They rarely use **val_bpb** as the *primary* objective; bits-per-dimension is common in generative modeling, but not as a *compression-aware deployment metric*.

2. **Neural compression & bits-back coding**  
   - Goal: compress data (images, text) using neural codecs, optimizing bits-per-pixel or bits-per-token.  
   - Gap:  
     - These works optimize **data compression**, not the **model artifact** itself.  
     - The 16MB constraint on the *entire inference package* (weights + runtime trickery) is not a typical design axis.

3. **Neural architecture search / hyperparameter tuning**  
   - Goal: maximize accuracy or minimize loss under FLOP/latency constraints.  
   - Gap:  
     - FLOPs/latency is different from *artifact size*; compression metadata and decompressor code are usually ignored.  
     - Quick-harness gating (short, standardized validation runs) as the *only* optimization signal for a parameter search over compression knobs is underexplored.

4. **LLM / LM compression with deployment constraints**  
   - Recent work on 4-bit/3-bit quantization, sparsity, KV-cache compression, etc., focuses on:  
     - Runtime memory and throughput, not *total artifact size* + val_bpb.  
     - Server or high-end mobile deployment, not “must be ≤16MB, end-to-end, and tuned by a tiny automatic search.”

**What’s timely now (2024–2026)**

Recent trends open a new opportunity:

- **Tiny / edge LMs and strict footprint constraints**  
  - There is rising interest in sub-20MB language models for on-device and browser deployment (e.g., WebLLM, tiny-LM variants).  
  - Tooling (e.g., GGUF, ONNX, WebGPU runtimes) makes it realistic to measure *full artifact size* precisely.

- **Val_bpb as a meaningful, stable metric for generative models**  
  - As language modeling gets more standardized, bits-per-token/byte is increasingly used as a robust metric beyond perplexity.  
  - A focus on val_bpb lets us connect model compression with **information-theoretic evaluation**.

- **Automated compression pipelines**  
  - Recent work on automated quantization and pruning shows that *small, structured hyperparameter searches* can yield large gains without huge compute.  
  - But these are not yet framed as **“parameter golf” under a strict artifact-size + quick-harness evaluation budget**.

**How this differs from standard approaches**

- **Objective**:  
  - Not “maximize accuracy at a given bit-width” or “minimize FLOPs”; instead:  
    > Minimize validation bits-per-byte **subject to** a 16MB *end-to-end* artifact budget, with each candidate evaluated through a fixed, fast harness.

- **Optimization object**:  
  - The *entire artifact*: quantization scheme, pruning mask, coding of weights, and decompressor implementation choices.  
  - The search explores **compression parameterizations** (bit-width schedules, grouping, entropy coding toggles, sparsity patterns), not only model hyperparameters.

- **Evaluation protocol**:  
  - A **quick-harness gate**: a standardized, limited-token validation pipeline (e.g., 50–200k tokens) that acts as the *only* inner-loop signal for the parameter search.  
  - This is different from long, full-corpus evaluation or training-time gradients.

- **Novel research question**:  
  - Can we design a *generic, low-compute, black-box “parameter golf” procedure* that reliably finds compression settings that:  
    - Fit within 16MB total artifact size, and  
    - Maintain val_bpb within a small tolerance of a much larger baseline model?
