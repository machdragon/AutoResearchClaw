# Stage 01: Topic Init

- **Topic**: Parameter Golf val_bpb minimization under quick-harness gate, with 16MB artifact compliance on promotion  
  *Domains*: machine-learning, efficient-training, model-compression  
  *Project*: `parameter-golf-autorc`  
  *Target quality*: ≥ 4.5 (reviewer average)

## Novel Angle

**Core idea**:  
Develop an *automatic parameter-golf controller* that jointly optimizes (a) validation bits-per-byte (val_bpb) and (b) a strict 16MB artifact size constraint, *under a quick-harness evaluation gate* (minutes-scale training/eval budget), by learning **structured, schedule-aware pruning/quantization policies** that are *conditioned on early-training signals* rather than final metrics.

This differs from standard compression / efficient training work in three ways:

1. **Quick-harness gate as a first-class constraint (early-signal–conditioned compression)**  
   - Most compression and efficient training papers assume:
     - Full training runs (many epochs) before evaluation.
     - Compression applied post hoc (after full training) or with full-epoch schedules.
   - Here, the *evaluation harness itself* is a constraint: models must be:
     - Trainable and evaluable in minutes.
     - Fully packaged (weights + tokenizer + config + harness glue) under **16MB**.
   - The method must make **compression decisions from early training curves** (e.g., first 1–3% of steps) to satisfy both:
     - Val_bpb target (compression quality).
     - Artifact size cap (deployment constraint).
   - This “quick-harness-gated compression policy learning” is not treated as a standalone research problem in existing work, which typically:
     - Uses long training schedules.
     - Ignores artifact size of the *full deployment bundle*.
     - Optimizes final loss/perplexity rather than val_bpb under a strict wall-clock and size gate.

2. **Joint optimization of val_bpb and *artifact-level* size (not just parameter count / model file)**  
   - Standard model compression focuses on:
     - FLOPs, parameter count, or model-file size.
     - Sometimes latency on specific hardware.
   - Here, the constraint is **“16MB artifact on promotion”**:
     - Includes model weights, tokenizer, config, and any runtime metadata required by the harness.
     - This forces *code-aware* and *tokenizer-aware* compression (e.g., vocab pruning, embedding factorization) rather than just weight pruning.
   - The research question:  
     **Can we learn a compression policy that reasons over *artifact composition* (weights + vocab + config) to minimize val_bpb under a hard 16MB bundle constraint?**
   - Existing “tiny language model” and “edge LLM” work rarely models the artifact as a compositional object; they generally report model size in parameters or checkpoint bytes, not *end-to-end deployable bundle size under a fixed harness*.

3. **Parameter Golf as a *search space*, AutoRC as a *controller***  
   - “Parameter Golf” (as used in small language model competitions / projects) is usually:
     - Manual or heuristic search over architectures, d_model, n_layers, vocab size, etc.
     - Sometimes accompanied by basic pruning/quantization.
   - The proposed angle:  
     - Treat “parameter golf” as a **discrete, constrained architecture–compression search space**.
     - Use **AutoRC** (automatic run configuration) style controllers:
       - Learn policies mapping early-run signals (e.g., first N steps of training loss, gradient norms, val_bpb trend) to:
         - Pruning schedules (layer-wise sparsity).
         - Quantization bitwidths per module.
         - Vocab / embedding factorization decisions.
       - Subject to a *hard 16MB artifact budget* and a *quick-harness wall-clock budget*.
   - Distinct from standard NAS / AutoML:
     - Search is *artifact-size and harness-time constrained*, not just accuracy-per-FLOP.
     - Objective is *val_bpb* (compression quality) on text, not generic accuracy.
     - Policies are **early-signal-conditioned** to be usable under quick-harness gate (no full-run evaluation for each candidate).

**Why timely (2024–2026)?**

- **Trend 1: Tiny language models and on-device LMs**  
  - There is a surge of work on <50MB and even <10MB LMs for phones, microcontrollers, and browsers.  
  - Competitions (e.g., TinyStories, micro-LLMs, LM competitions with strict size caps) and industry interest in *on-device assistants* make “artifact-size-constrained” training highly relevant.
