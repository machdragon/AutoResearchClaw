# Stage 08: Hypothesis Gen

Here is a synthesized set of final hypotheses that integrate the strongest ideas, address contrarian concerns, and stay feasible under your lane_3 constraints (int8+zlib, ≤16MB, ≤1.10× runtime). I keep explicit where perspectives still disagree.

## Hypothesis 1 – Compression-Contrastive QAT beats a *strong* post-hoc baseline by ≥1.5% relative post-roundtrip val_bpb

### Statement

When compared against a **well‑engineered float‑trained + post‑hoc int8+zlib baseline** (with strong calibration and minor structural tweaks), a **compression‑contrastive QAT regime**—treating the int8(+zlib‑proxy) path as a second view and enforcing multi‑layer invariance—will:

- Achieve **≥1.5% lower post‑roundtrip val_bpb (relative)** on held‑out validation,  
- While satisfying:  
  - final artifact size ≤16MB (int8+zlib),  
  - deployed runtime ≤1.10× the float baseline,  
- Under a fixed architecture, quantization scheme, and zlib configuration,  
- Within a single‑GPU, ≤30‑minute training budget for the small/medium models you’re targeting.

If no such gain appears after a modest but real hyperparameter sweep, this hypothesis is false.

### Rationale (innovation + pragmatism vs contrarian)

- **Innovator’s contribution (novelty):**
  - Go beyond vanilla QAT and KD by treating the **entire quant+compression transform as a second “view”** of the model:
    - Float view F: standard fp16/fp32 path.
    - Compressed view C: fake‑int8 + cheap zlib proxy path.
  - Enforce **multi‑layer view invariance** (BYOL-style feature regression + symmetric logit KD), explicitly pushing internal codes into a **compression‑stable subspace**.
- **Pragmatist’s constraints/feasibility:**
  - Use only:
    - Fake quant for weights (and optionally a small subset of activations).
    - A very cheap, differentiable-ish proxy for zlib (histogram/entropy proxy), not real zlib in the loop.
  - Restrict double forward to late training and to 1–2 bottleneck layers to stay under the 30‑minute budget.
- **Addressing contrarian Hypothesis A:**
  - The comparison baseline is not “naive PTQ” but a **strong post‑hoc pipeline** with:
    - per‑layer clipping / range search,
    - KL/entropy‑based calibration,
    - minor re‑scaling layers or bias corrections if allowed,
    - brief low‑LR fine‑tuning of quantized weights (without changing deployment graph).
  - Claim is **≥1.5% relative** improvement against this strong baseline, not just over a straw‑man.
- **Unresolved disagreement:**
  - Innovator/pragmatist expect a real gap if you optimize for the actual deployment transform.
  - Contrarian maintains that once you build a serious post‑hoc pipeline, any extra multi‑view complexity will yield <0.5% relative gains. This hypothesis directly tests that dispute.

### Method sketch (minimal viable, single GPU, ≤30 min)

1. **Baselines**

   Train under identical data, architecture, quant scheme, zlib config:

   - **Baseline 1: Strong post‑hoc int8+zlib**
     - Float training only.
     - After training:
       - Per‑layer clipping range search / calibration (e.g., minimizing quantization error or KL divergence).
       - Optional tiny scaling/centering layers that don’t change FLOPs materially.
       - Brief low‑LR fine‑tune directly on quantized weights (still with float forward).
     - Export to int8, compress with fixed zlib settings.
   
   - **Baseline 2: Standard late‑stage QAT (optional reference)**
     - Uniform fake‑quant turned on globally mid‑training, no multi‑view invariance, no zlib proxy.
     - Export to int8 + zlib as above.

2. **Proposed: Compression‑Contrastive QAT**

   Within the same wall‑clock budget:

   - Warmup (e.g., 40–50% of total steps): float‑only path F, standard task loss.
   - Late phase:
     - Two forwards per batch with **shared weights**:
       - Path F (float): compute task loss `L_task(F)`.
       - Path C (compressed proxy):
         - Fake‑int8 weights, optional fake‑int8 on 1–2 chosen bottleneck activations.
         - zlib proxy: histogram‑based entropy proxy or simple bit‑plane mask penalty, no backprop through real zlib.
     - Loss terms:
       - Task on F: `L_task(F)`.
       - Feature invariance at chosen bottleneck layers `h`:
         - `L_feat = || h_F – stopgrad(h_C) ||² + || stopgrad(h_F) – h_C ||²`.
       - Logit invariance:
         - Symmetric KD with temperature T:  
           `L_kd = KL(softmax(z_F/T) || softmax(z_C/T)) + KL(softmax(z_C/T) || softmax(z_F/T))`.
       - Optional compression proxy on weights:
         - `L_comp = λ · H_proxy(weights_int8)` where `H_proxy` ≈ entropy of quantized weight histograms.
     - Total loss:
       - `L = L_task(F) + α L_feat + β L_kd + γ L_comp`.
     - Only a modest grid on α, β, γ is allowed (e.g., {0.1, 0.3, 1.0}), to stay within the time budget.

3. **Constraints**

   - All models (baselines + proposed) must:
     - Compress with the same zlib config.
     - Produce artifacts ≤16MB.
     - Meet runtime ≤1.10× float on target hardware.
