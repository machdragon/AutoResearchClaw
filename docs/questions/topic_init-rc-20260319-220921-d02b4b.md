# Stage 01: Topic Init

- **Topic**:  
  Parameter-efficient compression of recurrent language models via LoRA specialization and lightweight routing/gating under strict artifact and runtime budgets (Parameter Golf val_bpb minimization).

---

### Novel Angle

This project targets a gap at the intersection of:

1. **Recurrent LMs (RNN-like / RWKV-style)**
2. **Parameter-efficient fine-tuning (PEFT, e.g., LoRA)**
3. **Mixture-of-experts / routing / gating**
4. **Ultra-tight deployment constraints** (explicit artifact size cap + runtime multiplier cap, evaluated via validation bits-per-byte, val_bpb)

What has been studied:
- **PEFT for Transformers**: LoRA and its variants are extensively explored for Transformers (e.g., LLaMA, GPT-style models) and for large-scale instruction tuning.
- **Mixture-of-Experts (MoE)**: GShard, Switch Transformer, DeepSeek-MoE, and many follow-ups analyze routing/gating in *transformer blocks* for throughput/quality trade-offs.
- **Recurrent architectures revival** (2023–2025): RWKV, Linear RNNs, Mamba/SSMs show that recurrent-style models can be competitive with Transformers for sequence modeling while being more deployment-friendly.
- **Model compression & distillation**: Quantization, pruning, and low-rank factorization are well-studied, mostly for Transformers and CNNs, often without *jointly optimizing* a precise artifact size budget and tight runtime constraint.

What is *not* (yet) well-studied – the core gap:
- **Composite recurrence + LoRA + lightweight routing/gating under *explicit* small-artifact + near-baseline-runtime constraints, evaluated by val_bpb.**
  - Existing PEFT work assumes a *separate* base model and adapter weights, often ignoring *total artifact size* as a hard cap (e.g., 16MB total deployable footprint) and focusing on parameter count or FLOPs instead.
  - Existing MoE and router work focuses on *compute scaling and quality* but not on:
    - tiny expert parameter budgets,
    - low-overhead gating compatible with *recurrent* architectures, and
    - the *runtime penalty* being strictly bounded (≤1.10× baseline).
  - Recurrent-specific compression work generally examines pruning/quantization or architecture tweaks, not **dynamic specialization via LoRA + gating** in a parameter-golf scenario.

Why this is timely (2024–2026):
- **Edge and local deployment trend**: Significant push toward on-device LMs (phones, micro-servers, local PCs), where hard caps on artifact size (e.g., 16MB) and slight runtime overheads are common.
- **Revived interest in recurrence/SSMs**: Work on RWKV, Mamba, and other linear-time sequence models provides competitive perplexities with lower memory and better streaming behavior. How to *specialize* these recurrent cores under tight parameter budgets is currently under-explored.
- **Mature but still-expanding PEFT ecosystem**: LoRA and derivatives have stabilized enough that *second-order design questions*—like where to place adapters, how to route to them, and how to trade size vs runtime vs quality—are now ripe for systematic exploration beyond Transformers.

Key differentiator from standard approaches:
- Instead of “MoE for big models” or “LoRA for Transformers,” this project studies **parameter-golf optimization in a recurrent LM**, where we:
  - start from a **recurrence-only baseline**;
  - **incrementally add** LoRA specialization layers and **residual gating/routing**;
  - enforce a **hard overall artifact budget (≤16MB)** and a **runtime quick-gate (≤1.10× baseline)**;
  - and measure improvements in **val_bpb** (bits-per-byte) on a standard byte-level LM benchmark.
- The novelty: targeting the **Pareto frontier of val_bpb vs. tiny artifact size vs. slight runtime overhead** in a *recurrent* setting using a **composite of LoRA + lightweight gating**, where almost all prior work either:
  - focuses on Transformer architectures,
  - ignores the strict artifact runtime/product constraints, or
  - does not use routing/gating with ultra-small experts.

---

### Scope

Focused for a single paper:

- **Model family**: A small to medium **byte-level recurrent LM** (e.g., RWKV-style or GRU/LSTM baseline) with fixed hidden size, trained from scratch or using a provided checkpoint.
- **Design space**:
  - Baseline: recurrence-only (no LoRA, no gating).
  - Variant A: add LoRA modules to a subset of recurrent weights.
  - Variant B: Variant A + simple scalar/vector residual gating that routes inputs between base and LoRA-augmented paths.
  - Variant C: explore minimal multi-expert gating (≤3 experts) under the same 16MB artifact budget.
- **Constraints**: Total parameter artifacts (base + LoRA + gating) ≤16MB; runtime overhead ≤1.10× baseline on a single GPU for typical batch/sequence lengths.
- **Outcome**: A clear, experimentally validated design recipe for **where and how** to inject LoRA and gating into recurrent LMs to minimize val_bpb, subject to strict deployment constraints.
