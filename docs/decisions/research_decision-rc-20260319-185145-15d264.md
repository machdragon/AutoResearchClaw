# Stage 15: Research Decision

## Decision
REFINE

## Justification

The minimum quality criteria for PROCEED are not met:

1. Only one (failed) run is available; there are **no valid baselines** with real metrics, and no reliable metric for the proposed method.
2. The *intended* primary metric (post–int8+zlib val_bpb, “lower is better”) is conceptually clear, but the values present are clearly **sentinels**, not real measurements, so the effective metric is undefined in practice.
3. Each condition has **n=1**; we do not have ≥3 seeds for anything.
4. Although the sentinel values differ in scale, ablation integrity is moot because the data are invalid and aggregation failed.
5. The analysis quality rating is explicitly **2/10**, below the ≥4/10 threshold.

Per the instructions: if any criterion fails, the decision must be REFINE, not PROCEED.  
Pivoting is not warranted: the core hypotheses and methodology are conceptually sound; the problems are in implementation and experimental rigor, not in the idea itself.

## Evidence

- Run status: **failed** with an exception in `statistics.pstdev` during aggregation.
- Metric magnitudes: e.g., `val_bpb ≈ 1e9`, `qat_cc/val_bpb_mean ≈ 1e8`, `_std = 0`, `n = 1` → clearly sentinel/placeholder values, not plausible outcomes.
- No completed, end-to-end **baseline** runs (float + PTQ + int8+zlib) with valid metrics.
- No multi-seed experiments (all metrics derived from a single run/seed).
- Explicit overall result-quality rating: **2/10**.
- Yet: the logging and constraint scaffolding (metric naming, QAT vs baseline symmetry, size/runtime constraints) is in place, suggesting the **hypothesis and design are fine**, but the plumbing is fragile.

## Next Actions

Focus on making the experimental pipeline trustworthy and instantiating a minimal valid study.

### 1. Fix Metrics and Aggregation Robustly

- Replace sentinel defaults with **NaN or explicit missing**:
  - If a metric cannot be computed, do not log a numeric placeholder like 1e9.
- Implement a `safe_aggregate(vals)` helper:
  - Filter to finite numeric values only.
  - If `len(vals) == 0`: return NaN for mean/std and mark the metric as unavailable.
  - If `len(vals) == 1`: return that value as `mean`, use NaN (or omit) for `std`, and **do not call** `statistics.pstdev`.
- Ensure:
  - Any run with an exception or incomplete pipeline sets `status = "failed"` and its metrics are **excluded from subsequent summaries/plots**.

### 2. Make the Roundtrip Metric Concretely Correct

- Explicitly log separate metrics for:
  - `val_bpb_float` (uncompressed float32 model).
  - `val_bpb_int8_posthoc` (float-trained → int8 quantized → zlib roundtrip → eval).
  - `val_bpb_qat_roundtrip` (QAT-trained → int8 → zlib roundtrip → eval).
- Verify the actual evaluation path is:
  - `float weights → int8 → zlib compress → zlib decompress → int8 weights → model eval`.
- Add:
  - `artifact_bytes` or `artifact_mb`.
  - `runtime_factor` vs a well-defined baseline.

### 3. Establish Strong, Explicit Baselines

Implement and verify at least:

- **B1:** Float32 model, no compression (reference only).
- **B2:** Float32 training + tuned PTQ → int8 + zlib roundtrip.

Ensure:
- Same training budget (epochs/steps, LR schedule, batch size) for B2 and QAT variants.
- Identical evaluation splits and procedures.

### 4. Run a Minimal “Micro” Experiment

Before scaling:

- Use a **small data subset** and short training schedule.
- Run:
  - B2 (float + PTQ + int8+zlib).
  - QAT-only variant.
  - Full QAT + proxy variant.
- Check:
  - Metrics are in plausible ranges (e.g., bpb in the right order of magnitude).
  - Artifact sizes and runtime are logged and sensible.
  - No aggregation crashes; NaNs only where genuinely missing.

Only once a micro experiment produces sane numbers should you move to full-scale runs.

### 5. Design the Full Evaluation Protocol

- Define the **primary metric** explicitly in code and documentation:
  - e.g., “Primary: `val_bpb_qat_roundtrip` (bits per byte, lower is better).”
- For each key configuration (B2, QAT variants):
  - Run **≥3 seeds**.
  - Aggregate with the robust `safe_aggregate`.
- Add minimal ablations:
  - PTQ only vs QAT only vs QAT + proxy vs full method.
- Make constraint metrics first-class:
  - Enforce/flag `artifact_mb ≤ 16` and `runtime_factor ≤ 1.10`.

Once these refinements are in place and you have valid, multi-seed baseline and QAT results, re-evaluate against the PROCEED criteria. At that point, PROCEED to paper-writing will be justified if the new runs satisfy all minimum quality constraints.
