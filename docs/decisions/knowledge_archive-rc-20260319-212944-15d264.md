# Stage 21: Knowledge Archive

## Parameter Golf Lane 3 Retrospective (PGQ, H1)  
Status: **REFINE (infrastructure + methodology), not PROCEED**

This archive summarizes what was attempted for lane_3 (“QAT + roundtrip-aware compression alignment”), what went wrong, what *tentative* signals we saw, and exactly what must change before the hypothesis can be re‑tested credibly.

---

## 1. Hypothesis & Setup (What We Tried)

**Lane:** Parameter Golf lane_3  
**Channel:** float → int8 weight pack → zlib → store → zlib decompress → int8 → run  
**Constraints:**

- Artifact cap: ≤ 16 MB
- Quick-gate: runtime_ratio ≤ 1.10 relative to a float baseline
- Metric: validation bits-per-byte *after* the full int8+zlib roundtrip (`val_bpb_roundtrip`)

**Hypothesis**

> Under the same runtime + size gate, a staged, compression‑aware QAT regime (PGQ) aligned to the int8+zlib roundtrip will yield *better post-roundtrip val_bpb* than a strong float-only baseline; a minimal entropy-regularized variant (H1) will further improve val_bpb_roundtrip.

**Conditions implemented (intended):**

1. `baseline_float_lane3`  
   - Standard float training, no QAT in-loop; deployment-only int8+zlib.

2. `QAT_only` (PGQ)  
   - Late-stage (“critical period”) fake int8 on weights in forward pass.  
   - No explicit entropy regularization.

3. `H1_minimal_self_organizing_QAT`  
   - Same staged QAT as `QAT_only`.  
   - Extra loss term favoring low-entropy histograms over quantized weight deltas (a proxy for zlib-friendly redundancy).

**Reported single-seed metrics (n = 1 each; not trustworthy for inference):**

- `baseline_float_lane3`  
  - `val_bpb` = 3.379443  
  - `val_bpb_roundtrip` = 3.424443  
  - `artifact_bytes` = 8,000,000  
  - `runtime_ratio` = 1.000  
  - `steps` ≈ 100k  

- `QAT_only`  
  - `val_bpb` = 3.358443  
  - `val_bpb_roundtrip` = 3.382068  
  - `artifact_bytes` = 4,800,000  
  - `runtime_ratio` = 1.124778 (violates ≤1.10)  
  - `steps` ≈ 90,240  

- `H1_minimal_self_organizing_QAT`  
  - `val_bpb` = 3.348843  
  - `val_bpb_roundtrip` = 3.369633  
  - `artifact_bytes` = 4,800,000  
  - `runtime_ratio` = 1.171261 (violates ≤1.10 even more)  
  - `steps` ≈ 87,352

## 2. What Actually Broke (Why We Can’t Trust the Results)

Several hard failures make these runs **non‑evidential**:

1. **Ablation integrity failures**

   - Automated checks flagged:  
     “ABRATION FAILURE … differentiating parameter is likely not used in the code.”
   - That implies: toggling `baseline` vs `QAT_only` vs `H1` did *not* reliably change the active computation or loss, at least for some key parameters.  
   - Causal statements like “QAT improved val_bpb_roundtrip” are therefore not supported.

2. **Harness / infrastructure failure**

   - The main run failed: `ModuleNotFoundError: No module named 'harness'`.
   - This means the current repository state does not execute a clean end-to-end lane_3 run. Metrics we see are from older or partial runs.

3. **n = 1 per condition, no variance**

   - Each of the three conditions has only one seed/run.
   - Typical LM variance at this scale is on the same order (or larger) than the observed ~0.02–0.05 bpb deltas.
   - No confidence intervals, no robustness checks.

4. **Inconsistent gate logic**

   - The “quick-gate” is defined as runtime_ratio ≤ 1.10.
   - Observed:  
     - `QAT_only` runtime_ratio = 1.1248 → flagged as gate fail (correct).  
     - `H1` runtime_ratio = 1.1713 → yet `quick_gate_passed = 1` in logs.  
   - That indicates either logging bugs or gate evaluation bugs, which undermines any budget-constrained comparison.

5. **Degenerate behavior across refine cycles**

   - Different “refine” attempts produced *identical* or near‑identical metrics (e.g., [3.35944261, 3.35844261]) despite configuration changes.
   - Strong sign that:
     - either metrics are cached / mis-logged, or
     - condition toggles aren’t wired into the actual training loop.

6. **Baseline inconsistencies**

   - Each condition logs its own “baseline” numbers, instead of all sharing a canonical float baseline. That confounds interpretation of any “ratio to baseline” metrics.

Net: the *idea* is intact, but the *platform and ablations* are currently unfit for hypothesis testing.

---

## 3. Lessons Learned

Despite the failures, several conceptual and practical lessons are clear.

### 3.1 Metric design is right; implementation is not

- Using `val_bpb_roundtrip` as the **primary endpoint** and explicitly tracking:
  - `artifact_bytes` (against 16 MB cap) and
  - `runtime_ratio` (against ≤1.10 gate)  
  is exactly the right way to operationalize lane_3.
- Splitting:
  - `val_bpb` (float or pre-roundtrip)
  - `val_bpb_roundtrip` (post-int8+zlib)  
  is the right structure to detect “channels where QAT pays off only after roundtrip.”

What failed is implementation:

- Non-deterministic or inconsistent baselines.
- Gates applied incorrectly.
- Metrics that may be reading from stale checkpoints or caches.
